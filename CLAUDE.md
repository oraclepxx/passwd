# AGENTS.md ‚Äî Implementation Guide for passwd

This file defines the rules, constraints, and ordered steps for implementing the passwd app.
Reference `TECH_DESIGN_v2.md` for all architectural decisions. Do not deviate from them without explicit instruction.

---

## Rules

### General
- Always read this file and `TECH_DESIGN_v2.md` before starting any implementation work.
- Implement only what is specified. Do not add features, abstractions, or "nice to haves" not listed in the design docs.
- Never modify `AGENTS.md`, `PRD.md`, `TECH_DESIGN_v2.md`, or `RESEARCH.md` during implementation. These are source-of-truth documents.
- After completing each phase, run the verification steps defined at the end of that phase before proceeding.

### Testing
- Every code change must trigger a full test run before the change is considered complete.
- Before running tests, reset all items in `specs/TEST_TASKS.md` from `[x]` to `[ ]` and update the Coverage Summary statuses to `‚¨ú Pending`.
- Run the backend test suite (`GO111MODULE=on go test ./backend/... -v`) and the frontend test suite (`cd frontend && npm test`) in full.
- After both suites finish, mark each item in `specs/TEST_TASKS.md` as `[x]` (pass) or `[~]` (fail/skip) based on actual results, and update the Coverage Summary.
- Write a `specs/TEST_REPORTS.md` file after every test run. It must include: the date/time, the commit or change description, per-suite pass/fail counts, and a section listing every failed or skipped test with its failure message.
- If any test fails, do not mark the change as complete. Fix the failure and re-run the full suite before proceeding.

### Task Execution
- Implement phase by phase, in the order defined in `TASKS.md`. Never start a new phase before the current one is fully complete and verified.
- Before starting any new phase, read `TASKS.md` and check that every task (`[ ]`) in all previous phases is marked done (`[x]`). If any prior task is not done, stop and complete it before proceeding.
- Within each phase, implement task by task in the order listed. Do not jump ahead or batch tasks across phases.
- After completing each individual task, immediately mark it as done in `TASKS.md` by changing `[ ]` to `[x]`.
- After all tasks and verify steps in a phase are marked done, stop and report the phase summary to the user. Wait for explicit instruction before starting the next phase.

### Go Backend
- All crypto operations must stay in the Go backend. Never perform encryption, decryption, or key derivation in the frontend.
- Use Go standard library for all crypto: `crypto/aes`, `crypto/cipher`, `crypto/rand`. Do not add third-party crypto packages.
- Use `crypto/rand` for all randomness (nonces, salt, password generation). Never use `math/rand`.
- All public methods on the `App` struct must call `a.vault.IsUnlocked()` and return an error immediately if the vault is locked. The only exceptions are `VaultCreate`, `VaultUnlock`, and `VaultIsUnlocked`.
- Zero sensitive byte slices (vault key, master password bytes) immediately after use. Use `clear(slice)` or a manual loop.
- Return Go `error` values from all App methods. Never panic on user-triggered errors.
- Use `database/sql` with `modernc.org/sqlite` driver. Never use CGo-based sqlite3.

### Frontend
- Call the Go backend exclusively through the auto-generated Wails bindings in `wailsjs/go/`. Never use `fetch`, `XMLHttpRequest`, or any direct network call.
- Never store a plaintext password in React state longer than necessary to render it. Clear it when the user navigates away from the detail view.
- Passwords are never shown by default. Reveal only on explicit user action (click).
- All destructive actions (delete, permanent delete) must show a `ConfirmDialog` before proceeding.
- `wailsjs/` is auto-generated by Wails ‚Äî never edit files inside it manually.

### Security
- `search_hint` must contain only `name` and `username`, lowercased, tab-separated. Never include password, URL, or notes.
- Every encryption call must generate a fresh 12-byte nonce via `crypto/rand.Read`. Never reuse nonces.
- Clipboard must be cleared after the configured timeout (default 30 seconds). This must be implemented in Go, not the frontend.
- Wails CSP must deny all external origins. No outbound network requests are permitted.

### Code Style
- Go: follow standard `gofmt` formatting. Use named return values only when they aid clarity.
- TypeScript: use strict mode (`"strict": true` in tsconfig). No `any` types.
- React: functional components with hooks only. No class components.
- Tailwind: utility classes only, no custom CSS files unless absolutely necessary.

### Phase 3 ‚Äî Crypto Layer (Go)

**Goal:** Key derivation and AES-256-GCM encrypt/decrypt functions.

**Steps:**
1. Implement `backend/vault/crypto.go`:
   - `DeriveKey(password []byte, salt []byte) ([32]byte, error)` ‚Äî Argon2id with params `m=65536, t=3, p=4`.
   - `Encrypt(key [32]byte, plaintext []byte) (ciphertext []byte, nonce []byte, error)` ‚Äî AES-256-GCM, fresh nonce per call.
   - `Decrypt(key [32]byte, ciphertext []byte, nonce []byte) ([]byte, error)` ‚Äî AES-256-GCM verify + decrypt.
2. Implement `backend/vault/session.go`:
   - `VaultSession` struct holding `key [32]byte` and `unlocked bool`, protected by `sync.Mutex`.
   - `Lock()` method: zeros `key` with `clear(s.key[:])`, sets `unlocked = false`.
   - `IsUnlocked() bool`, `Key() ([32]byte, error)`.

**Verify:**
- Unit test: encrypt then decrypt a known plaintext, assert round-trip equality.
- Unit test: tamper with ciphertext, assert `Decrypt` returns an error.
- Unit test: `Lock()` zeroes the key ‚Äî assert `key == [32]byte{}` after lock.
- `go test ./backend/vault/...` passes.

### Phase 4 ‚Äî Vault & Record Business Logic (Go)

**Goal:** All CRUD and vault management logic, ready to be wired to the App struct.

**Steps:**
1. Implement `backend/vault/records.go` with these functions (each takes `db *DB` and `key [32]byte`):
   - `CreateRecord(db, key, input RecordInput) (string, error)`
     - Generate UUID v4 for `id`
     - Build `search_hint` = `strings.ToLower(name + "\t" + username)`
     - Encrypt `RecordPlaintext` JSON ‚Üí store ciphertext + nonce
   - `ListRecords(db, key, query string) ([]RecordSummary, error)`
     - Filter by `search_hint LIKE ?` if query non-empty
     - `WHERE deleted_at IS NULL`
     - Apply `maskUsername` to each result (see `TECH_DESIGN_v2.md` ¬ß5)
   - `GetRecord(db, key, id string) (RecordDetail, error)` ‚Äî decrypt on demand
   - `UpdateRecord(db, key, id string, input RecordInput) error`
     - If password changed: insert old password into `password_history`, prune to 5 (SQL from ¬ß8)
     - Re-encrypt full record with fresh nonce
   - `DeleteRecord(db, id string) error` ‚Äî sets `deleted_at = now()`
   - `RestoreRecord(db, id string) error` ‚Äî sets `deleted_at = NULL`
   - `PurgeRecord(db, id string) error` ‚Äî hard DELETE, requires `deleted_at IS NOT NULL`
   - `GetHistory(db, key, id string) ([]PasswordHistory, error)` ‚Äî last 5, decrypt passwords
   - `PurgeExpiredTrash(db) error` ‚Äî delete where `deleted_at < now() - 30 days`

2. Implement `maskUsername` in `records.go` exactly as specified in `TECH_DESIGN_v2.md` ¬ß5.

3. Implement `backend/vault/generator.go`:
   - `Generate(opts GeneratorOptions) (string, error)`
   - Character sets: lowercase always included; symbols, numbers, uppercase based on opts
   - Guarantee at least one character from each enabled set
   - Fill remaining length from full pool via `crypto/rand`
   - Fisher-Yates shuffle using `crypto/rand`
   - Return error if `opts.Length < 8`

**Verify:**
- Unit test `CreateRecord` + `GetRecord` round-trip: decrypted values match input.
- Unit test `ListRecords` with query filter: only matching records returned.
- Unit test `UpdateRecord` with password change: history row inserted, old value recoverable.
- Unit test `maskUsername`: verify all cases from `TECH_DESIGN_v2.md` ¬ß5.
- Unit test `Generate`: assert length, assert each enabled character class present, assert no `math/rand` used.
- `go test ./backend/vault/...` passes.

### Phase 5 ‚Äî App Struct & Wails Wiring (Go)

**Goal:** All bound methods on `App` struct wired to vault logic.

**Steps:**
1. Implement `backend/app.go` ‚Äî `App` struct with fields:
   ```go
   type App struct {
       ctx     context.Context
       db      *vault.DB
       session *vault.VaultSession
   }
   ```
2. Implement `startup(ctx)` and `shutdown(ctx)` lifecycle methods:
   - `startup`: open DB, run `PurgeExpiredTrash`
   - `shutdown`: call `session.Lock()`, close DB
3. Implement all bound methods from `TECH_DESIGN_v2.md` ¬ß4.2. Each method that operates on records must:
   ```go
   if !a.session.IsUnlocked() {
       return ErrVaultLocked
   }
   key, _ := a.session.Key()
   ```
4. Implement `ClipboardCopy(value string, timeoutSecs int) error`:
   - Write to OS clipboard
   - Start a goroutine that sleeps `timeoutSecs` seconds then clears the clipboard
5. Register `App` in `backend/main.go` with `wails.Run`.

**Verify:**
- `wails build` produces a binary with no errors.
- Run the binary; confirm the app window opens.
- In Wails dev tools console: call `window['go']['main']['App']['VaultIsUnlocked']()` ‚Äî assert it returns `false`.

### Phase 6 ‚Äî Frontend: Vault Lock Screen

**Goal:** Functional lock/unlock/create vault UI.

**Steps:**
1. Implement `LockView.tsx`:
   - On first launch (`VaultIsUnlocked` returns false and no vault exists): show "Create vault" form with password + confirm fields.
   - On subsequent launches: show "Unlock vault" form with single password field.
   - Call `VaultCreate(password)` or `VaultUnlock(password)` on submit.
   - Show inline error on wrong password.
   - On success: transition to `ListView`.
2. Implement `useVault.ts` hook:
   - Tracks `isUnlocked: boolean`
   - Exposes `unlock(password)`, `lock()`, `createVault(password)`
   - Auto-lock: reset a `setTimeout` on `mousemove` and `keydown`; call `lock()` on expiry (default 5 min)
3. Wire `App.tsx` to render `LockView` when locked, main layout when unlocked.

**Verify:**
- Create a vault with a password; confirm app transitions to empty list view.
- Lock the vault (call `VaultLock` from console or wait for auto-lock); confirm lock screen reappears.
- Enter wrong password; confirm error message shown.
- Enter correct password; confirm unlock succeeds.

### Phase 10 ‚Äî Security Hardening & Final Checks

**Goal:** Verify all security requirements from `TECH_DESIGN_v2.md` ¬ß12.

**Steps ‚Äî work through the checklist one item at a time:**

1. **Vault key zeroing:** Add a test that locks the vault and asserts `session.key == [32]byte{}`.
2. **Nonce uniqueness:** Code review `crypto.go` ‚Äî confirm `crypto/rand.Read` used for every nonce, no nonce reused.
3. **search_hint safety:** Code review `records.go` `CreateRecord` and `UpdateRecord` ‚Äî confirm `search_hint` never includes password, URL, or notes.
4. **Locked vault guard:** Code review all `App` methods ‚Äî confirm every record method returns `ErrVaultLocked` if session is locked.
5. **CSP:** Open `wails.json`; confirm `CSP` field denies all external origins (e.g., `"default-src 'self'"` with no external URLs).
6. **Clipboard auto-clear:** Manual test ‚Äî copy a password, wait 30 seconds, assert clipboard is empty.
7. **Trash auto-purge:** Write a test that inserts a record with `deleted_at = now() - 31 days`, calls `PurgeExpiredTrash`, asserts record is gone.
8. **No `math/rand`:** Run `grep -r "math/rand" backend/` ‚Äî assert no results.
9. **No outbound network:** Run the app with network monitoring (e.g., `lsof -i` or Little Snitch); assert no external connections made on startup or normal use.

**Verify:**
